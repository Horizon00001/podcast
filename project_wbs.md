# 项目工作分解结构（WBS）

## 一、后端基础设施（FastAPI）

| 任务编号 | 任务名称 | 职责描述 | 产出物 | 依赖 |
|---------|---------|---------|-------|------|
| 1.1 | 后端框架搭建 | 搭建 FastAPI 项目结构，配置数据库连接（SQLite/PostgreSQL），定义基础数据模型（播客、用户、交互记录等） | 项目骨架、数据模型定义 | 无 |
| 1.2 | API 接口开发 | 实现播客列表、播客详情、用户行为上报（播放/点赞/收藏）、推荐列表获取、手动生成播客触发等接口 | RESTful API 文档、接口代码 | 1.1 |
| 1.3 | 数据存储与访问层 | 实现播客数据、用户数据、交互数据的 CRUD 操作，设计用户行为表结构 | DAO 层代码、数据库迁移脚本 | 1.1 |
| 1.4 | 系统部署与演示环境 | 配置 Docker 或直接部署，提供可访问的演示地址 | 部署脚本、演示环境 | 1.2, 2.3, 3.3, 4.2 |

## 二、资讯聚合与脚本生成

| 任务编号 | 任务名称 | 职责描述 | 产出物 | 依赖 |
|---------|---------|---------|-------|------|
| 2.1 | RSS 抓取与数据清洗 | 使用 feedparser 从指定 RSS 源抓取新闻，提取标题、正文、发布时间等，进行文本清洗（去除HTML标签、特殊字符） | RSS 抓取模块代码 | 无 |
| 2.2 | Prompt 工程与脚本生成 | 设计系统 Prompt，调用 DeepSeek API 将新闻文本重构为双人对话脚本（含角色标识、分段标记） | 脚本生成模块代码、Prompt 模板 | 2.1 |
| 2.3 | 内容特征提取 | 从生成的脚本中提取主题词、关键词、实体等，用于推荐模块的内容特征 | 特征提取接口、特征数据存储 | 2.2 |
| 2.4 | 脚本输出与格式规范 | 定义脚本输出的 JSON 格式（包含文本内容、角色、时间戳占位符等），供音频模块和前端使用 | 脚本格式规范、示例文件 | 2.2 |

## 三、音频合成与后期处理

| 任务编号 | 任务名称 | 职责描述 | 产出物 | 依赖 |
|---------|---------|---------|-------|------|
| 3.1 | 通义千问 TTS API 调用 | 封装调用通义千问 TTS API 的代码，支持多角色、多音色、情感控制 | TTS 调用模块代码 | 无 |
| 3.2 | 音频拼接与混音 | 使用 pydub/FFmpeg 将多个角色音频片段按时间顺序拼接，混入开场白、背景音乐、过渡音效 | 音频后期处理模块代码 | 3.1, 2.4 |
| 3.3 | 时长控制与音质优化 | 调整 TTS 语速、停顿，确保每期播客时长在 5-8 分钟；对音频进行标准化处理 | 优化后的音频文件、参数调优记录 | 3.2 |
| 3.4 | 本地 TTS 部署探索（可选） | 调研并尝试本地部署开源 TTS 模型（如 VITS、ChatTTS 等），提供可行性方案文档 | 可行性报告、实验代码 | 3.1 |

## 四、个性化推荐引擎

| 任务编号 | 任务名称 | 职责描述 | 产出物 | 依赖 |
|---------|---------|---------|-------|------|
| 4.1 | 用户行为数据收集 | 设计前端埋点方案，后端接收并存储用户行为（播放、点赞、收藏、跳过等），构建用户-物品交互矩阵 | 埋点代码、行为数据存储逻辑 | 1.2 |

## 任务依赖关系总结

### 基础先行
- 后端框架（1.1）和数据模型是所有模块的基础

### 生成流程
- RSS 抓取（2.1）→ 脚本生成（2.2）→ 音频合成（3.1-3.3），形成核心播客生成流水线

### 推荐依赖
- 用户行为数据收集（4.1）需前端埋点（5.4）配合
- 内容特征提取（2.3）依赖脚本生成（2.2）
- 热度推荐（4.2）可独立实现
- 协同过滤（4.3）需积累一定数据

### 前端联动
- 前端需后端接口（1.2）支持
- 高亮功能需脚本格式（2.4）和音频时间戳（3.2）配合

### 收尾工作
- 测试与文档在核心功能完成后集中开展
